---
title: Connect to a remote Host
description: Configure SSH setup or direct HTTPS access to a Host Daemon.
---

Satelle supports two authenticated remote paths:

- `ssh` uploads and starts an on-demand Host Daemon through system OpenSSH.
- `direct` connects to an already-running Host Daemon over HTTPS and WSS.

Both paths require a pinned Host identity and a file-backed Satelle API token.
SSH authenticates the transport, but never replaces Satelle API authentication.

## Set up an SSH Host

Create an owner-controlled user configuration with an SSH destination, the
expected Host identity, and an absolute token-file destination:

```toml
default_host = "studio-mac"

[hosts.studio-mac]
transport = "ssh"
adapter = "codex"
address = "operator@studio-mac.example.ts.net"
expected_host_id = "host-018f-example"

[hosts.studio-mac.api_token]
kind = "file"
path = "/absolute/private/path/satelle-token"
```

The token file must not already exist for the first setup. Its parent directory
must be owner-only. Later setup reruns validate and reuse that file without
replacing it. Run a transport-only dry-run first, then apply with explicit
consent:

```sh
satelle setup --host studio-mac --on-demand --component transport --dry-run --json
satelle setup --host studio-mac --on-demand --component transport --yes
```

Setup starts a loopback-only bootstrap daemon, asks the Host to generate a
short-lived pending credential, atomically publishes the raw token to the
configured owner-only file, then activates the durable credential. If
verification fails, setup revokes the pending credential and removes the new
token file after the Host confirms it inactive. Unactivated pending credentials
also expire automatically.

Later `run` and `steer` commands start an unavailable on-demand daemon without
another bootstrap token and authenticate with the durable token file.

## Start the Host Daemon

For a loopback-only Host, bind the server explicitly:

```sh
satelle host start --bind 127.0.0.1:3001 --foreground
```

The default bind address is loopback. Exposing the daemon beyond loopback
requires a TLS certificate and owner-only private key in addition to the Host
identity and token boundary described below:

```sh
satelle host start --bind 0.0.0.0:3001 \
  --tls-cert /absolute/path/to/host-cert.pem \
  --tls-key /absolute/private/path/to/host-key.pem \
  --foreground
```

## Configure direct HTTPS

Direct HTTPS/WSS connections are supported with an operator-provisioned,
file-backed token. Automatic direct-token provisioning is not implemented in
this MVP: the SSH setup flow above cannot mint or provision credentials for a
direct binding.

Create an owner-controlled user configuration with an HTTPS address, expected
Host identity, token file descriptor, and CA bundle:

```toml
default_host = "studio-mac"

[hosts.studio-mac]
transport = "direct"
adapter = "codex"
address = "https://studio-mac.example.ts.net:3001"
ca_bundle = "/absolute/path/to/satelle-ca.pem"

[hosts.studio-mac.api_token]
kind = "file"
path = "/absolute/path/to/satelle-token"

[hosts.studio-mac.network]
provider = "tailscale"
tailnet_name = "example.ts.net"
hostname = "studio-mac"
```

The token file must be owner-only. The CA bundle is explicit trust material,
not an instruction to disable certificate validation.

Tailscale can provide the routable name and private network path, but it does
not replace Satelle authentication, TLS validation, or Host identity pinning.

Trust the observed Host identity interactively after verifying the endpoint and
certificate:

```sh
satelle host trust --host studio-mac
```

After confirmation, Satelle writes the observed `expected_host_id` into the
owner-controlled user configuration.

For noninteractive trust, require the exact identity supplied out of band:

```sh
satelle host trust --host studio-mac \
  --expected-host-id <exact-host-id> --yes --no-input
```

## Verify before running work

```sh
satelle config check --host studio-mac --json
satelle host status --host studio-mac --json
satelle run --host studio-mac --detach --json "Open the browser"
```

For SSH Hosts, the same verification commands use the authenticated tunnel and
the durable token created by setup.

Direct setup is not implemented. Provision the direct token, TLS certificate,
private key, and CA bundle through operator-controlled tooling before using
`host trust`.

# Specification Gaps

This ledger records behavior that existing facts do not define and that blocks implementation or verification. It must not add product scope or substitute for `.facts`.

## Open Gaps

### GAP-004 - Release attestation trust policy

Description: Release facts require attestations and verification but do not define the complete trust policy.
Component: Release attestation and verification
Exposing fact or work unit: Attestation generation, archive verification, TUF trust roots, and signed-tag verification
Undefined behavior: Predicate type, archive versus raw-binary subject, verifier mechanism, TUF bootstrap and root-update policy, and the signed-tag trust source.
Blocking: yes

### GAP-005 - Installer and package-manager lifecycle contract

Description: Distribution facts do not close the native installer and package-manager lifecycle across supported platforms.
Component: Native installers, Homebrew, Scoop, npm, pnpm, and Bun distribution
Exposing fact or work unit: Installer validation and release publication gates
Undefined behavior: Canonical URLs, arguments, destinations, receipt schema and lifecycle, upgrade and uninstall semantics, failure atomicity, Linux and Windows 11 architecture matrix, tap and bucket identities, ownership detection, and release sequence.
Blocking: yes

### GAP-006 - Embedded skill bundle CLI contract

Description: Skill-bundle facts do not define the observable CLI and materialization contract.
Component: Embedded agent skill distribution
Exposing fact or work unit: `skills get`, `skills path`, version output, and public command examples
Undefined behavior: Output formats, path materialization semantics, bundle membership including `SKILL.md` and `agents/openai.yaml`, version output, and command-example scope.
Blocking: yes

### GAP-007 - Recovery suggestion command contract

Description: Facts do not establish that every emitted recovery suggestion is a complete executable command.
Component: CLI, MCP, setup, doctor, and completion recovery output
Exposing fact or work unit: Canonical `suggested_commands` verification
Undefined behavior: Whether prose is permitted, the canonical replacement for invalid `host trust`, supported doctor storage scope, and whether completion placeholders may be emitted as commands.
Blocking: yes

### GAP-008 - Public documentation and support boundary

Description: Public documentation facts do not close the version, deployment, support, and vulnerability-reporting contract.
Component: README, Fumadocs, SECURITY.md, and release documentation
Exposing fact or work unit: Public documentation and release-readiness gates
Undefined behavior: Authoritative release version, versioned routes and deployment target, Node.js floor, public source roots, private vulnerability-reporting channel, and Windows 11 support boundary.
Blocking: yes

### GAP-009 - CLI process exit-code class taxonomy

Description: MVP error facts define a small broad process exit-code taxonomy, while completion contracts and the current error mapping emit additional stable numeric classes.
Component: CLI parsing, typed errors, process exits, and error conformance
Exposing fact or work unit: Canonical parser and semantic error conformance, `ErrorCode::exit_code`, completion writes, storage integrity, Host lookups and contention, TLS, authentication and authorization, readiness and state conflicts, and not-implemented command paths
Undefined behavior: Whether the classes named by `1gy` are exhaustive; whether completion exit 73 is an explicit exception or part of an expanded taxonomy; the canonical classes for storage integrity, TLS, authentication and authorization, contention, lookup failures, state conflicts, and not-implemented paths; and which failures must use internal exit 70.
Blocking: no - parser-format selection and the shared human and `satelle.error.v1` envelope can proceed, but final numeric normalization and exhaustive exit-code conformance must wait for this decision.

### GAP-010 - Release archive integrity and root-layout contract

Description: Release facts require a canonical SHA256SUMS manifest and archive validation, but do not fully define the checksum manifest syntax, the digest subject checked by release validation, or the allowed archive member layout beyond one root executable.
Component: Release archive construction, integrity metadata, and prepublication validation
Exposing fact or work unit: Facts `9t9`, `2s8`, `3ha`, `9qc`, and `h9h`; release archive generation; SHA256SUMS generation and parsing; deterministic archive validation
Undefined behavior: SHA256SUMS line grammar, separator or binary marker, hex case, filename escaping, line endings, entry ordering, and duplicate, missing, or extra entry handling; whether `9qc` validates the compressed archive bytes, the extracted executable bytes, or both; and whether `3ha` permits non-executable files, directories, links, metadata-only entries, or duplicate archive members in addition to the single root executable.
Blocking: yes - lower-level archive reading, SHA-256 computation, and the policy-neutral `3ha` single-executable check can proceed, but the canonical manifest generator, strict full-archive validator, and implementation claims for `2s8` and full `9qc` require these decisions.

### GAP-011 - Versioned payload versus process-channel conformance

Description: Fact `4ze` requires stdout, stderr, and exit-status coverage for every versioned JSON and NDJSON surface, but public HTTP and WebSocket payloads do not have process channels or process exit status.
Component: CLI, MCP, HTTP, WebSocket, and shared wire-contract verification
Exposing fact or work unit: Fact `4ze` and the closed schema, enum-token, framing, and channel matrix
Undefined behavior: Whether `4ze` applies only to CLI and MCP process output, or whether transport payloads are also in scope with HTTP status, content type, WebSocket frame type, and close behavior replacing process-only stdout, stderr, and exit status requirements.
Blocking: yes - existing surface-specific tests may be strengthened, but `4ze` cannot be tagged implemented without choosing the intended scope and native channel equivalents.

# Specification Gaps

This ledger records behavior that existing facts do not define and that blocks implementation or verification. It must not add product scope or substitute for `.facts`.

## Open Gaps

### GAP-001 - Token administration replay contract

Description: Fact `x7h` requires durable duplicate issuance or rotation behavior, but the frozen facts do not define a public token-administration operation.
Component: Host and transport API token administration
Exposing fact or work unit: `x7h`, with `wmh`, `vq7`, `lqg`, `p9x`, `992`, `d0n`, `05z`, and `lcn`
Undefined behavior: Exact methods, routes, request and response schemas, durable metadata fields, fresh and replay HTTP statuses, result versus error semantics, and recovery fields.
Blocking: yes - Phase 2 cannot implement `x7h` without inventing a public contract.

### GAP-002 - Canonical route-set contradiction

Description: Implemented fact `sh84` includes `/v1/setup` and `/v1/doctor`, while `eo5` calls them future routes, `uqk` excludes them from the current operation set, and the router exposes neither route.
Component: Host Daemon public API and capabilities
Exposing fact or work unit: `sh84`, `eo5`, and `uqk`
Undefined behavior: Whether setup and doctor are current public routes or reserved future routes.
Blocking: yes - route and capability conformance cannot prove both claims.

### GAP-003 - Queue envelope lifecycle fields

Description: Later queue facts do not close the queue status, cancellation, result, and event envelope before and after admission.
Component: Durable queued-turn API
Exposing fact or work unit: `c54`, `vzg`, and `i6x`
Undefined behavior: Revision field names and presence, closed queue states, normalized failure shape, identifier nullability before admission, and the transition that links admitted Session and Turn identifiers.
Blocking: no - the affected facts are later work, not current MVP work.
